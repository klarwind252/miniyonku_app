# 14.14 GWファーム：サーバー応答への対処（2026-07-23 確定）

M4LAPS は**クラウド版かつライセンス登録済みの環境でのみ利用可**。
サーバー側は全エンドポイントを `require_m4laps` で保護し、条件を満たさない場合は
**404** を返す（403 ではなく 404 なのは、機能の存在自体を隠すため）。

対象（12エンドポイント・すべて保護済み）:

| 系統 | エンドポイント |
|---|---|
| GW受信口 | `POST /api/timing/races`／`POST /api/timing/races/{id}/events` |
| ノード割当 | `POST /api/timing/join`／`GET /api/timing/assignments` |
| PIP | `GET /api/timing/pip/latest` |
| 管理画面 | `/admin/timing/results`（詳細・削除含む）／`/devices`／`/layouts` ほか |

## DA16. GWは 404/401 を「恒久的な拒否」として扱う（★ファーム実装時の必須仕様）

サーバーが 404 を返すのは「このサーバーでは M4LAPS が使えない」という意味であり、
**何度送り直しても結果は変わらない**。通信エラー（一時的な失敗）と区別すること。

```
POST の結果
 ├ 200/201 → 成功。保持していたデータを削除する
 ├ 404     → M4LAPS無効（オンプレ版／ライセンス未登録／サーバーが古い）
 │            ⚠ 再送しない・蓄積しない・以降の送信を停止
 │            TFTに「サーバー未対応」を表示し、運営が気づけるようにする
 ├ 401     → トークン不一致（設定ミス）。404と同じ扱い
 │            TFTに「認証エラー」を表示
 └ その他 / タイムアウト / 接続失敗
              → 一時的な失敗。保持して後で再送（R4の後送りに従う）
```

**なぜこの区別が必要か**

1. **無限リトライの防止** — 404を一時失敗と誤解して再送し続けると、電池を消耗し
   通信路も無駄に使う。計測そのものには影響しないが、当日のバッテリー持ちに響く。
2. **LittleFSの枯渇防止** — 送信できないぶんを溜め続けると容量が尽きる。
   404は「今後も送れない」ので**溜める意味がない**。R4の保持上限とは別に、
   そもそも保持対象から外す。
3. **原因の可視化** — TFTに何も出ないと、運営はWiFiや電波を疑って時間を溶かす。
   「サーバー未対応」と出れば、アプリ側の設定を確認すればよいと分かる。

⚠ 計測（ESP-NOW・打刻）は**サーバーと無関係に動き続ける**こと。
   送信できないだけで計測を止めてはいけない。記録はLittleFSに残し、
   サーバーが使える環境に繋ぎ直したときに送れる状態を保つ
   （ただし404の場合は溜めない＝上記2のとおり）。

## DA17. ベスト記録はサーバー側で保持する（GWは関与しない）

その日のベスト・レース内ベストは、**サーバーが受信時に更新して保持**する
（`timing_bests` テーブル）。GWは送信するだけでよく、ベスト判定の責務を持たない。

- **受信時**（`POST /events` 完了時）に、そのレースと当日のベストを更新
- **削除時**にその日のベストを再計算（消した記録がベストのまま残らないように）
- 期間指定など任意条件のベストは保持せず、必要時に別途集計する
  （リアルタイム性が不要なため）

保持する指標は6つ。タイム系は最小・速度系は最大が「ベスト」。

| metric | 内容 | 良い方向 |
|---|---|---|
| total | トータルタイム | 最小 |
| lap | ラップタイム | 最小 |
| sector | セクタータイム | 最小 |
| max_ms | 最高速度 | 最大 |
| lap_avg | ラップ平均速度 | 最大 |
| sector_ms | セクター通過速度 | 最大 |

⚠ 速度（`max_ms` / `sector_ms`）と平均速度（`lap_avg`）は、
   **ビーム間隔・コース全長の設定が未実装のため現状はダミー値**。
   実装時は `_dummy_speed_ms` / `_dummy_lap_avg_ms` を実測値に差し替える
   （計算式は各関数のdocstringに記載）。
